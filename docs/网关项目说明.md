# AI Gateway 项目说明

本文说明 **AI Gateway**（`com.example.aigateway`）的定位、架构与使用方式，便于接入方（如 ops-agent）与运维同学快速理解。

## 1. 项目定位

AI Gateway 是一个 **自研 Spring Boot 3** 的 **多厂商 LLM 统一入口**：

- **对外**：仅暴露 **OpenAI Chat Completions** 兼容接口：`POST /v1/chat/completions`（请求/响应 JSON 形态与 OpenAI 一致，便于现有客户端零改或微改）。
- **对内**：按路由把请求转发到 **OpenAI**、**DeepSeek**（OpenAI 兼容透传）、**Anthropic Messages API** 或 **Google Gemini generateContent**；各厂商 **API Key 只存在于网关进程**（环境变量 / 密钥管理），业务应用只持有 **网关客户端令牌**。

典型拓扑：

```text
[业务应用: ops-agent] --Bearer(客户端令牌)--> [AI Gateway] --各厂商密钥--> [OpenAI / DeepSeek / Anthropic / Gemini]
```

## 2. 代码与模块结构

工程为单模块 Maven 应用，主要包：

| 包路径 | 职责 |
|--------|------|
| `config` | `GatewayProperties`（客户端令牌、各上游 baseUrl/apiKey）；`WebClient` 超时；`GatewayBackendsConfiguration`（OpenAI/DeepSeek 透传 Bean） |
| `proxy` | `LlmBackend` 接口；`OpenAiPassthroughBackend`；`AnthropicBackend` + `AnthropicTranslator`；`GeminiBackend` + `GeminiTranslator`；`BackendRegistry`（按头/model 选后端） |
| `service` | `ChatProxyService`：解析 body、拒绝 `stream:true`、套 Resilience4j 限流后调用后端 |
| `security` | `ClientTokenAuthFilter`：校验 `Authorization: Bearer` 是否在 `GATEWAY_CLIENT_TOKENS` 白名单（空则开发模式放行） |
| `web` | `ChatCompletionsController`；`ApiExceptionHandler`（上游 4xx/5xx 透传、限流 429 等）；`GatewayRequestLogFilter`（访问日志，不落 Authorization 全文） |

**可观测**：Spring Boot Actuator（`health`、`prometheus`、`metrics`）。

## 3. 请求处理流程

1. **鉴权**：除 `/actuator/**` 外，请求需带 `Authorization: Bearer <客户端令牌>`（若配置了 `GATEWAY_CLIENT_TOKENS`）。
2. **入口**：`ChatCompletionsController` 接收原始 JSON 字符串 body，读取可选头 **`X-Route-Provider`**。
3. **路由**：`BackendRegistry.resolve(header, body)`  
   - 若 `X-Route-Provider` 为 `openai` / `deepseek` / `anthropic` / `gemini`，直接使用对应后端。  
   - 否则按 **`model` 字段** 推断：`deepseek-*` → DeepSeek；`claude*` 或含 `anthropic` → Anthropic；`gemini*` 或 `google/*` → Gemini；否则 → OpenAI。
4. **转发**：  
   - **OpenAI / DeepSeek**：`OpenAiPassthroughBackend` 使用 `WebClient` 将 **同一 JSON body** POST 到上游 `{baseUrl}/v1/chat/completions`，并替换为上游 `Authorization: Bearer <上游密钥>`。  
   - **Anthropic**：将 OpenAI 的 `messages`（含 system）转换为 Anthropic **`/v1/messages`** 请求体，调用后将响应 **再包装成 OpenAI chat.completion 形状** 返回客户端。  
   - **Gemini**：将 messages 转为 **`generateContent`** 请求体，请求 URL 含 `?key=`，响应同样 **包装成 OpenAI 形状**。
5. **限流**：`ChatProxyService.proxyChatCompletions` 使用 Resilience4j **`gatewayChat`** 实例（默认每分钟请求数可配）。

## 4. 配置要点（环境变量）

完整列表见仓库根 [README.md](../README.md)。核心如下：

- **`GATEWAY_CLIENT_TOKENS`**：逗号分隔的客户端 Bearer；**生产必须非空**。为空时网关打 WARN 并放行（仅本地开发）。
- **`UPSTREAM_*`**：各厂商 baseUrl 与 API Key，彼此独立。
- **`GATEWAY_RATE_LIMIT_PER_MINUTE`**：全局限流阈值。

## 5. 与 ops-agent（a1）的集成

ops-agent 侧：

- `OPENAI_BASE_URL` 指向网关根地址（无尾斜杠），例如 `http://localhost:8090`。
- `OPENAI_API_KEY` 使用 **网关颁发的客户端令牌**（与 `GATEWAY_CLIENT_TOKENS` 中某一项一致）。
- 可选 **`OPSAGENT_LLM_OPENAI_ROUTE_PROVIDER`**：映射为请求头 **`X-Route-Provider`**，显式指定 `openai|deepseek|anthropic|gemini`，避免仅靠 `model` 推断歧义。

上游真实密钥 **不要** 出现在 ops-agent 配置中。

## 6. 已知限制与后续扩展

- **流式**：当前若 body 中 `stream: true`，返回 **400 `stream_not_supported`**。后续可在网关内做 SSE 透传或分厂商流式适配。
- **Gemini**：当前实现针对 **Google AI Studio（API Key + generativelanguage.googleapis.com）**；**Vertex AI**（服务账号、区域端点）需在 `GeminiBackend` / 配置中增加分支。
- **Anthropic**：当前为 **非流式** Messages 调用；工具调用（tool_use）与流式事件需在翻译层逐项对齐官方文档后扩展。

## 7. 构建与测试

```bash
mvn test
```

单测覆盖：Anthropic/Gemini JSON 转换、`BackendRegistry` 路由、`OpenAiPassthroughBackend` 与 MockWebServer 联调、`stream:true` 拒绝逻辑等。CI 见 `.github/workflows/ci.yml`。

## 8. 安全建议

- 生产环境务必配置 **`GATEWAY_CLIENT_TOKENS`**，并定期轮换。
- 日志中避免打印完整 Bearer；上游错误体可截断记录。
- 网关与上游之间使用 TLS；密钥来源推荐 Vault / 云 KMS，而非明文进镜像。
